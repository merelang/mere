# Mere on Cloudflare Workers — Snippet Share

Cloudflare Worker sample that dogfoods the Mere → CF Worker pipeline
with a real reason to exist: a playground-snippet share service.
POST raw code, get a short URL; anyone visiting the URL sees the code.

Compiled wasm: ~8 KB. State lives in Cloudflare KV. Runtime deps:
zero (the wasm's env stubs everything Node-specific).

## Endpoints

| Route              | Behaviour |
|--------------------|-----------|
| `GET /`            | Landing HTML with usage |
| `POST /share`      | Body = raw code → returns `{"id","url"}` and stores in KV |
| `GET /s/:id`       | Returns stored snippet as plain text (404 if unknown) |

## What's here

- `main.mere` — Mere source. Dispatches on method + path, generates
  8-hex IDs via `gen_request_id`, returns response JSON. Uses two
  externs (`cf_body` / `cf_kv_lookup`) to receive request body and
  pre-fetched KV value out-of-band (avoids JSON escape round-trip
  pitfalls for arbitrary code text).
- `worker.js` — CF Worker entry (ES module). Provides all externs,
  handles the async KV plumbing around the sync Mere call.
- `wrangler.toml` — CF Worker deploy config, including a KV
  namespace binding placeholder.
- `build.sh` — compiles Mere → wat → wasm.
- `local_test.js` — Node smoke test with an in-memory KV mock.
- `main.wat` — committed for reference (regenerated by `build.sh`).

## Design: async KV, sync Mere

Cloudflare KV bindings are Promise-based (`env.MERE_SNIPPETS.get(k)`
returns a Promise). Mere externs are synchronous. To bridge:

- **Pre-fetch** (read path): For `GET /s/:id`, JS awaits
  `KV.get(id)` BEFORE calling the Mere handler. The result lives in
  a module-scoped `currentKvLookup` variable and Mere reads it via
  the `cf_kv_lookup ()` extern.
- **Outbox** (write path): For `POST /share`, Mere's response JSON
  includes `"kv_put":{"key":..., "value":...}`. JS honours this
  directive AFTER the Mere handler returns, via `KV.put()`.

Body handling follows the same "extern-not-JSON" convention:
request body goes into `currentBody`, retrieved by Mere via
`cf_body ()`. This sidesteps a subtle bug where an earlier
JSON-in-JSON encoding doubled `\n` sequences.

## Building

    sh build.sh

Produces `main.wasm` (~8 KB).

## Local smoke test

    node local_test.js

Six assertions: landing page, POST /share stores + returns id,
GET /s/:id retrieves the original code byte-for-byte, 404 on
unknown id, 400 on empty body, 404 on unknown route.

## Deploying to Cloudflare

Prerequisites: `wrangler` CLI (`npm install -g wrangler`), a
Cloudflare account with Workers enabled, and `wrangler login` done.

1. Create the KV namespace:

        wrangler kv namespace create MERE_SNIPPETS

   Copy the returned `id` into `wrangler.toml` (replace
   `REPLACE_WITH_KV_NAMESPACE_ID`).

2. Build + deploy:

        sh build.sh
        wrangler deploy

3. Test the deployed worker:

        curl -X POST --data-binary 'let x = 1 in x + 1' \
          https://mere-share.<your-cf-subdomain>.workers.dev/share
        # → {"id":"...","url":"/s/..."}

`wrangler dev` runs the same worker locally in a V8 isolate with
emulated KV:

    wrangler dev            # → http://localhost:8787

## Why this needs Cloudflare Workers (not just Pages)

The direction paper's principle: don't force-fit CF Worker where a
static site suffices. This sample IS a genuine Worker use case:

- **Server-side state** — snippets live in KV, not in the browser
- **Global low-latency** — reads served from the nearest CF edge
- **API endpoint** — a static site can't accept `POST /share`
- **Small compute** — random ID generation, request dispatch, JSON
  assembly

GitHub Pages or Cloudflare Pages (static-only mode) could serve the
`GET /` landing page but nothing else on this list. The write path
(`POST /share` → KV) is where Workers earn their keep.

## What DOESN'T work on Cloudflare Workers

CF Workers run in V8 isolates without Node.js APIs. Unavailable in
this environment:

- **TCP** (`tcp_connect` / `tcp_read` / `tcp_write`) — no raw
  sockets. Use `fetch()` for outgoing HTTP if needed.
- **Subprocess** (`subprocess_run`) — no `child_process`.
- **File I/O** (`read_file` / `write_file`) — no `fs`. Static assets
  go through the wrangler `[assets]` binding.
- **Sync sleep** (`sleep_ms`) — `Atomics.wait` isn't allowed on the
  main isolate loop.

For KV, R2, D1 use CF-provided bindings via the pattern in
`worker.js`.
