# Operations runbook

Operational procedures for running the Mere project — building and
deploying the docs site, managing merelang.org, deploying the
Cloudflare Workers. This is the "HOW", with the footguns called out.
Design rationale ("WHY") lives elsewhere.

Not published to the docs site (the SSG only walks `docs/`).

---

## Docs site: build & deploy

The site at <https://merelang.org/> is a static site generated by a
Mere program (`contrib/site/build.mere`) — the SSG dogfoods the
compiler.

### Local build

```sh
sh contrib/site/build_full.sh docs _site
```

This runs:
1. `contrib/site/build.mere` — markdown → HTML + `style.css` + `index`
   + `search.json` + `sitemap.xml` + `.nojekyll`, and copies
   `contrib/site/playground/*.{html,wat}`.
2. Regenerates the self-host playground `.wat` bridges (fmt / repl /
   tyck / compile) from their `.mere` sources.
3. `wat2wasm --enable-tail-call` on every playground `.wat`. The
   `--enable-tail-call` flag is REQUIRED — contrib demos with `while`
   / inner-lifted closures emit `return_call[_indirect]` (Wasm
   tail-call proposal). Without the flag, `wat2wasm` rejects them.
4. Copies `contrib/dom/dom.glue.js` next to the playground HTML.
5. Writes `_site/CNAME` = `merelang.org` (custom domain — see below).

Serve locally: `cd _site && python3 -m http.server 8765`.

### Auto-deploy

`.github/workflows/pages.yml` runs on push to `main`: builds via the
same `build_full.sh`, uploads `_site` as a Pages artifact, deploys
via `actions/deploy-pages@v4`. `build_type` is `workflow` (Actions),
NOT the legacy "deploy from branch".

### Transient deploy failures

The `deploy` job occasionally fails with
`Deployment failed, try again later.` — this is a GitHub Pages
platform-side transient, not a repo problem (the `build` job succeeds).
Fix: re-run just the failed job.

```sh
gh run rerun <run-id> --repo merelang/mere --failed
```

If it fails 2-3× in a row, wait a few minutes and retry — it always
clears. The next successful push also supersedes it.

### Link paths — why root serving works

All generated links are RELATIVE (`changelog.html`,
`playground/counter.html`, `style.css` — no leading slash, no `/mere/`
prefix). So the site serves identically whether at the old
`merelang.github.io/mere/` (project subpath) or the current
`merelang.org/` (apex root). Don't introduce root-absolute (`/foo`)
or `/mere/`-prefixed links — they'd break one context or the other.

`sitemap.xml` uses relative `<loc>` (suboptimal SEO, not broken —
a follow-up could bake in the absolute base URL).

## merelang.org: DNS + custom domain + HTTPS

Deployment topology (decided 2026-07-07):

| domain | serves | cloud |
|--------|--------|-------|
| `merelang.org` (apex) | docs site (GitHub Pages) | **grey (DNS only)** |
| `www.merelang.org` | → apex (GitHub auto-redirect) | grey (DNS only) |
| `pkg.merelang.org` | package registry (CF Worker) | orange (proxied) |
| `api.merelang.org` | snippet share (CF Worker) | orange (proxied) |

DNS is hosted at Cloudflare (nameservers `*.ns.cloudflare.com`).

### First-time setup (already done — reference for a new domain)

1. Add the domain to Cloudflare, change registrar nameservers to the
   two CF nameservers, wait for the zone to show **Active**.
2. Cloudflare DNS records, **grey cloud (DNS only)**:
   - apex: `CNAME  @  →  merelang.github.io` (CF CNAME-flattening
     makes an apex CNAME legal)
   - `CNAME  www  →  merelang.github.io`
3. `build_full.sh` emits `_site/CNAME` = `merelang.org` on every build.
   With the Actions-based deploy this file is **inert** until the
   custom domain is explicitly set in step 4 (verified: a deploy with
   the CNAME artifact but no repo config left `gh api .../pages`
   `cname=null` and the old URL served 200, no redirect). So it's safe
   to pre-stage.
4. Set the custom domain:
   ```sh
   gh api --method PUT repos/merelang/mere/pages -f cname='merelang.org'
   ```
   (or Settings → Pages → Custom domain in the UI). This makes GitHub
   route the domain + start Let's Encrypt cert provisioning.
5. Wait for the cert (minutes to ~24h). When ready:
   ```sh
   gh api --method PUT repos/merelang/mere/pages -F https_enforced=true
   ```
6. Verify (bypass any local DNS negative-cache with `--resolve`):
   ```sh
   IP=$(dig merelang.org +short | head -1)
   curl -sS -o /dev/null -w "%{http_code}\n" --resolve merelang.org:443:$IP https://merelang.org/
   ```

### ⚠️ FOOTGUN: apex must stay grey cloud (never orange)

GitHub Pages renews its Let's Encrypt cert every ~90 days via an
HTTP-01 challenge to `/.well-known/acme-challenge/`. If the apex is
**orange cloud (Cloudflare proxy)**, CF intercepts that path and the
renewal FAILS — the site keeps working until the cert expires, then
HTTPS breaks with no warning (a delayed landmine).

**Rule: apex + www stay grey cloud (DNS only).** Only the Worker
subdomains (`pkg.` / `api.`) go orange — those ARE Cloudflare Workers,
so they belong on CF's proxy anyway. The docs site loses nothing
meaningful (GitHub/Fastly already CDN it).

### Recovery: HTTPS broke after ~90 days

If `https://merelang.org/` starts serving cert errors:

1. Confirm the apex is grey cloud in Cloudflare (if someone flipped it
   to orange, that's the cause — flip back to grey).
2. Force GitHub to re-provision: clear + re-set the custom domain.
   ```sh
   gh api --method PUT repos/merelang/mere/pages -f cname=''
   # wait ~1 min
   gh api --method PUT repos/merelang/mere/pages -f cname='merelang.org'
   # wait for cert, then:
   gh api --method PUT repos/merelang/mere/pages -F https_enforced=true
   ```
   (Same off/on toggle as Settings → Pages → remove/re-add domain.)

### ⚠️ FOOTGUN: CNAME ordering

Set up DNS **before** activating the custom domain. If you set the
repo custom domain while DNS doesn't yet resolve to GitHub, the
`*.github.io/mere/` URL 301-redirects to a non-resolving `merelang.org`
→ live site broken during propagation. (With Actions deploys the
artifact CNAME alone is inert, so pre-staging the file is fine — it's
the `gh api ... cname=` / Settings step that flips the switch.)

## Cloudflare Workers: registry & snippet share

Two Worker samples live in `examples/`:

- `examples/cloudflare-worker-registry/` → `pkg.merelang.org`
  (package registry v0.1, read-only JSON API)
- `examples/cloudflare-worker/` → `api.merelang.org` (snippet share,
  KV-backed)

They are TEMPLATES in `examples/` today. Promote to their own repo
when running them as production services.

### Build + deploy a Worker

```sh
cd examples/cloudflare-worker-registry   # or cloudflare-worker
sh build.sh                              # main.mere → main.wat → main.wasm
node local_test.js                       # smoke test (no CF account needed)
wrangler deploy                          # needs wrangler CLI + `wrangler login`
```

### KV setup (snippet share only)

```sh
wrangler kv namespace create MERE_SNIPPETS
# copy the returned id into wrangler.toml's [[kv_namespaces]] block
```

### The async-KV ↔ sync-Mere bridge

Mere externs are synchronous; CF KV bindings are async. Two
conventions bridge them (see `examples/cloudflare-worker/worker.js`):
- **Pre-fetch (read)**: JS `await KV.get()` BEFORE calling the Mere
  handler, stashes the value in module scratch, Mere reads it via a
  `cf_kv_lookup ()` extern.
- **Outbox (write)**: Mere returns `kv_put:{key,value}` in its
  response JSON; JS honours it AFTER the handler returns via `KV.put`.

Request body follows the same "extern-not-JSON" convention (`cf_body
()`) to avoid double-escaping arbitrary text through the JSON envelope.

### What doesn't work on CF Workers

V8 isolate, no Node stdlib. Unavailable Mere externs: TCP
(`tcp_connect`/…), subprocess (`subprocess_run`), file I/O
(`read_file`/`write_file`), sync sleep (`sleep_ms` — `Atomics.wait`
disallowed on the isolate loop). Use `fetch()` for outgoing HTTP, CF
bindings for storage.

## Running the test suite

```sh
dune test              # 1849+ tests
dune test --force      # bypass dune's cache if a run reports 0 tests
```

CI (`.github/workflows/ci.yml`) runs the suite on ubuntu + macOS ×
OCaml 5.1 / 5.4 (4-matrix). Both `ci.yml` and `pages.yml` must be
green before a release.
